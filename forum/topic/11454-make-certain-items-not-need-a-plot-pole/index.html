        The requirement is set to 1 by default (i.e. you need them) for everything and then set to 0 via the items class definition for items that do not need them (generators, workbench etc).Â  The classes are in cfgVehicles.hpp (GitHub version here) but the gotcha is that you cannot download them with mission files.Â  They have to be in the @mod download.

        Â 

        The alternative is to setup an array in the scripts with the new values and instead of checking in the config files for the DZE_requireplot entry, just check the custom array and set it based on a match.Â  I am doing this for fuel capacity as a lot of the class files do not have it set.

        Â 

        Setting 

        DZE_requireplot = 0;

        in the init.sqf will default everything to 0 (not needed).

        fn_selfactions.sqf and fn_damageactions.sqf setup the scroll menu items.Â  They are scanned continually to find a matching condition as a player moves around (is the cursor target XXX then add a new item YYY which, if chosen calls script ZZZ).

        Â 

        You could put a bit of code in there that scans for nearest object plot poles, check the owner id against the players id andÂ checks the owners ID against the friendlies ids.Â  If no match is found for either does a remove action on every action marked as addedÂ (s_[variable for action item] = 1).Â  Both sqf scripts would have to do the same (make it a checkNearPlot.sqf function).

        Â 

        THe problem is that if the scripts run every second (I have no idea how often they are fired but prety rapidly) then the find neaest plot pole will take a bit of time, comparing will take a bit of time, checking every actionitem flag will take a bit of time and removing items will take a bit of time so I would expect a noticable slowdown.

        Another way may be to create a trigger when the plot pole is created which would then do the check and removal when the trigger zone is entered (plot pole range + 5 meters would probably be best).Â  That would be much better on resources.Â  Would probably need to set an abort flag for the selfactions and damageactions scripts as well or they will be fighting to add menu items and the trigger will be fighting to remove them...
